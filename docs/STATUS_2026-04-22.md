## Why zero trades

Same DOGE long setup is being re-generated by the 1H formation detector every 5-min scan. The engine's deterministic gates (HTF, level, phase, retest, etc.) pass. Agent pulls its 14-day track record, sees the C|sideways profile is net-negative, vetoes. Next scan: repeat.

This is **not a bug.** It's:
- 82 instances of the agent correctly refusing to re-take a losing pattern
- Cost ≈ $2 / $250-per-month-projected for defence against a statistically-losing profile
- Previously these would have been 82 × ~$10-25 losing scratches = potentially $800–2000 in losses

The DOGE setup will either:
- Invalidate structurally (formation ages out, level changes) → different setup → different decision
- Eventually produce a profile with a differentiator the agent notices → one APPROVE
- Stay in this pattern for days until regime changes (crypto range-bound markets can persist)

## What to watch for (green / yellow / red)

### ✅ Green (healthy)
- New `mm_agent_decisions` rows arriving every ~5 min during active sessions
- `prompt_version = 'prompt_v=2 rubric_v=2'` on all recent rows
- VETO reasons cite specific profile stats (e.g. *"C|sideways profile is 1W/4L"*)
- `concerns` arrays include `recent_losses` when Rubric 8 fires
- Cost staying under $20/day

### ⚠️ Yellow (tune)
- Same setup re-evaluated >200 times in 24h with identical outcome → add a short-circuit cache
- Monthly cost projection crosses $400 → consider caching or tightening
- Agent approves a NEW profile → check next day whether it won/lost; iterate rubric if systematically wrong
- No decisions logged for >1h during a normally-active session → agent pipeline stalling

### 🔴 Red (investigate immediately)
- `decision='ERROR'` rate >10% over an hour → Anthropic API / key issue
- Zero decisions for 24h+ → scan pipeline broken before agent gate
- Approve rate spikes to >80% on same profile that's in `recent_losses` → rubric regression / prompt cache served stale content
