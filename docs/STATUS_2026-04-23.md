## Batch A — Rubric 8 min-n filter (`eda4193`)

### What was broken

Yesterday's STATUS framed the 0% approve rate as *"flat + learning > losing"*. Re-diagnosis today proved that generous. The Tier 2 learning loop was citing n=1 and n=5 profile buckets to veto setups — statistically these are noise, not signal. 235 setups vetoed in 48h, 1 approved, 0 traded.

### Diagnosis (captured 2026-04-22 22:29 UTC)

```
Health:           HTTP 200 (bot alive, v44)
Last MM trade:    2026-04-21 00:05 UTC (~46h ago)
Last agent call:  2026-04-22 11:55 UTC (~10.5h ago — funnel-reject upstream)
```

Decisions by day:

| Day | APPROVE | VETO | ERROR | Approve rate |
|---|---|---|---|---|
| 2026-04-20 (pre-v44) | 7 | 9 | 1 | 41% |
| 2026-04-21 (v44 shipped) | 1 | 133 | 0 | 0.7% |
| 2026-04-22 | 0 | 102 | 0 | **0%** |

Representative VETO reasons on 2026-04-22 (verbatim):

> *"Rubric 5: Grade F with 2/4 retest is weakest allowed; **Rubric 8: C\|sideways 1W/4L/-$159 over 6 samples** signals regime against us."*

> *"Rubric 5/8: Grade F same-session three_hits_how with 2/4 retest; **F\|sideways profile already 0W/1L** and C\|sideways 1W/4L show regime hostile to shorts."*

n=1 ("0W/1L") and n=6 ("1W/4L") are well within variance of any underlying win rate. Treating them as regime signal is over-fitting. Combined with selection bias (the buckets are only filled by the agent's own approvals), `rubric_v=2` created a one-way ratchet — every loss ratcheted the agent further into veto-everything.

### What shipped

- `src/config.py` — new `mm_sanity_agent_outcome_min_n: int = 20`. Tunable via `MM_SANITY_AGENT_OUTCOME_MIN_N` env var or `engine_state.config_overrides` in Supabase (no redeploy).
- `src/strategy/mm_sanity_agent.py` — `_build_user_prompt` filters outcome stats to buckets with `wins + losses + scratches >= min_n`. Empty → "(insufficient data …)" line instructing the model to skip Rubric 8.
- `PROMPT_VERSION` bumped `prompt_v=2 rubric_v=2` → `prompt_v=3 rubric_v=3`.
- 6 new tests in `TestUserPromptOutcomeStatsFilter`, 3 existing updated.

### What to watch for after deploy

**✅ Green (fix working):**
- `prompt_version = 'prompt_v=3 rubric_v=3'` on new `mm_agent_decisions` rows
- At least **some** APPROVEs on valid Grade A/B/C aligned-HTF setups within 12-24h
- VETO reasons no longer citing n<20 profile stats
- The phrase *"insufficient data"* appearing in input_context where Rubric 8 is skipped

**⚠️ Yellow (partial):**
- Approve rate still <5% after 24h across >50 decisions → Rubrics 1-7 are the real blocker, not Rubric 8
- A bucket reaches n=20 and is net-negative → designed behaviour; verify the bucket genuinely represents a losing regime

**🔴 Red (regression):**
- Zero approves over 48h AND every VETO cites "insufficient data" → filter not working
- `cache_read_input_tokens = 0` on every call → SYSTEM_PROMPT hash drifting
- Approve rate jumps to >60% immediately → filter accidentally disabled

### Known limitations (Batch A)

1. **Selection-bias doom loop (partial).** min_n=20 delays the trap but doesn't remove it — once a bucket reaches n=20 and is losing, the ratchet re-engages. Real fix: two-way signal (positive buckets surface as bullish prior too). ROADMAP §new.
2. **Rollback docs-as-truth of STATUS_2026-04-22.** Yesterday's framing ("flat + learning > losing") was generous given what the agent was actually doing. Kept as historical record.

## Batch B — User-reported bugs (P1 + P2 + P3)

After the first week of live operation, user flagged three concrete failures. Each got a clean root cause, a course citation, and a regression test.

| # | Commit | Summary |
|---|--------|---------|
| P1 | `01038af` | Entry-slippage gate from peak2 wick (Lesson 20/47) |
| P2 | `a2c00d1` | Per-setup agent decision cache (30-min TTL, 0.5% drift) |
| P3 | `40a0860` | MFE-based 2h scratch rule (Lesson 13 "within" = window) |

### P1 — entry-slippage gate

**Before:** `entry_price = current_price` (whatever tick just arrived). On 2026-04-20 the BTC setup had a peak2 wick at $72,000 but the engine recorded entry at $75,251 — 4.39% slippage. That pushed SL 12.47% below entry (vs course-correct 8.45%), which then tripped the "SL too wide" reject. Rejection reason was a symptom, not the cause.

**After:** new `mm_max_entry_slippage_pct` config (default 1.0%). If current_price has moved >1% past the peak2 wick, setup is rejected with `rejection_reason='entry_slipped_from_retest'`. The course-correct entry is the wick, not the tick.

### P2 — agent decision cache

**Before:** 82 identical Opus 4.7 calls on the same DOGE long setup in 6 hours. The 1H formation detector re-emits the same setup every 5-min scan, each re-emit ≈ $0.05. Projection: $250/month (designed for $6).

**After:** in-memory cache, key = `(symbol, direction, variant, round(entry, 4dp))`, TTL = 30 min, drift threshold = 0.5%. Cached verdicts report zero cost/latency and prefix reason with `[cached]`. Kill switch short-circuits BEFORE cache lookup (no stale replays after disabling the agent). Malformed responses are not cached.

### P3 — MFE-based scratch rule

**Before:** At 2h elapsed, if `gross_pnl <= round_trip_fees`, scratch. Measured ONLY the current snapshot. User's complaint: *"if the initial trade goes in the wrong direction, but comes back in the direction we expect, it gets closed before it can actually realized."* Course Lesson 13 says "substantial profit WITHIN two hours" — that's a window.

**After:** track Max Favorable Excursion in R-multiples on every manage tick. Scratch only if peak MFE never cleared `mm_scratch_mfe_threshold_r` (default 0.3R) during the 2h window. A trade that peaked at +0.5R and retraced to break-even survives. MFE is persisted to `trades.mm_max_favorable_excursion_r` (migration 020) on meaningful increases, restored on restart.

### Config knobs added by Batch B

| Flag | Default | Effect |
|------|---------|--------|
| `mm_max_entry_slippage_pct` | 1.0 | Reject setups where price slipped >1% past peak2 wick. 0 disables. |
| `mm_sanity_agent_cache_ttl_seconds` | 1800 | Cache lifetime. 0 disables the cache. |
| `mm_sanity_agent_cache_price_drift_pct` | 0.5 | Drift above which cache invalidates within TTL. |
| `mm_scratch_mfe_threshold_r` | 0.3 | Peak MFE required (in R) to exempt from 2h scratch. |

All tuneable via `engine_state.config_overrides` (runtime) or env var (startup).

### Migration 020 — required before deploy

```sql
ALTER TABLE trades ADD COLUMN IF NOT EXISTS mm_max_favorable_excursion_r numeric DEFAULT 0;
```

See `migrations/020_mm_scratch_mfe.sql`.
