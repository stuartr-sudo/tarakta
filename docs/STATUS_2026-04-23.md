## Headline

**v44 was over-vetoing, not "correctly refusing." Today's fix ships `rubric_v=3` with an engine-side min-n filter on Rubric 8.**

Yesterday's STATUS framed the 0% approve rate as *"flat + learning > losing"*. Re-diagnosis today proved that generous. The Tier 2 learning loop was citing n=1 and n=5 profile buckets to veto setups — statistically these are noise, not signal. 235 setups vetoed in 48h, 1 approved, 0 traded.

The fix pre-filters the outcome block in the user prompt: buckets need ≥20 closed samples to be shown. Rubric 8 text rewritten to explicitly tell the model to SKIP this rubric when its own profile isn't in the (filtered) block — closing the "model cites data the rubric said to ignore" hole.

## What shipped today

### `src/config.py`
New setting:
```python
mm_sanity_agent_outcome_min_n: int = 20
```
Tunable via `MM_SANITY_AGENT_OUTCOME_MIN_N` env var or `engine_state.config_overrides` in Supabase (no redeploy).

### `src/strategy/mm_sanity_agent.py`

`_build_user_prompt` now filters outcome stats:
```python
eligible = {
    k: v for k, v in outcome_stats.items()
    if (wins + losses + scratches) >= min_n
}
```
When `eligible` is empty (common case until buckets accumulate), the block renders:
```
  (insufficient data — no profile has >=20 closed samples yet;
   skip Rubric 8 and decide on Rubrics 1-7 only)
```

Rubric 8 rewritten to instruct: *"If the block shows '(insufficient data ...)' OR the current setup's profile is NOT listed: SKIP this rubric entirely. ... Never cite a profile that isn't in the block."*

`PROMPT_VERSION` bumped `prompt_v=2 rubric_v=2` → `prompt_v=3 rubric_v=3`. System-prompt cache rebuilds on first call post-deploy (~$0.04 one-time cache write).

### `tests/test_mm_sanity_agent.py`

Three existing tests updated: `test_no_stats_renders_insufficient_data_placeholder` (new unified message), `test_stats_rendered_with_this_profile_marker` (sample sizes bumped to realistic n≥20 values), `test_prompt_version_bumped_to_rubric_v3` (renamed + updated).

## What is NOT in today's fix

2. **Problem B from diagnosis** — 10h with nothing reaching the agent. BTC three_hits_how with RR=0.04 re-scans every 5 min. Engine-side issue (narrow TP1 on same-session three_hits targeting its own peak). Logged separately.
3. **Rollback docs-as-truth of STATUS_2026-04-22.** Yesterday's framing ("flat + learning > losing") was generous given what the agent was actually doing. The file stays as the historical record of what was believed on that day, but today's STATUS supersedes it.
